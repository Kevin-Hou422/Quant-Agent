# 因子研究与实盘交易 — 长期指导原则

**制定日期：** 2026-07-20
**性质：** 长期参考文档。记录经得起时间考验的定位、原则与判据，
不含具体实现细节（实现进度见 `PAPER_TRADING_ROADMAP.md`）。
**修订规则：** 只有当某条原则被实际运营证伪时才修改，并保留修订记录。

## 一、因子挖掘在实盘体系中的定位

### 1.1 价值链位置

```
数据层 → [因子挖掘] → 验证门 → 组合构建 → 执行 → 风控 → 归因反馈
          （候选生成）  （筛选）   （资本分配） （落地） （保命） （闭环）
```

**因子挖掘是候选生成器，不是决策者。** 它处于整条价值链的最上游，
职责是持续产出"有经济逻辑、统计上未被证伪、彼此低相关"的候选信号流。
盈利不由挖掘环节决定，而由下游三件事决定：验证纪律（杀掉假阳性）、
组合构建（资本分配与风险预算）、执行质量（成本控制）。

### 1.2 正确的衡量指标

- **错误指标**：单因子回测 Sharpe（受多重检验膨胀，且不含下游损耗）
- **正确指标**：
  - 候选存活率：CANDIDATE → 通过 60 日 paper 验证的比例
  - 池多样性：存活因子间的平均相关性与家族分布
  - 谱系可信度：任一因子可完整回答"它如何被选出、经过哪些检验"
- 行业事实：挖掘产出的绝大多数候选会死在验证门，这是系统健康的标志
  而非失败。存活率异常高反而说明验证门太松。

### 1.3 当前工作模式的符合度与演进方向

当前系统的定位是正确的（GP 产出 → Pool 去相关 → 生命周期门槛），
差距在形态：**交互式研究工具 → 无人值守候选流水线**。

目标形态：收盘后调度器自动触发挖掘 → 候选自动过统计验证门
（WalkForward 全折正 + DSR 阈值）→ 合格者自动进 CANDIDATE 并开始
paper 跟踪 → **人只在 PAPER→ACTIVE 的资本分配决策点介入**。
人的稀缺注意力应花在"是否给真钱"，不应花在"盯着 GP 跑"。

## 二、过程透明度原则：可审计谱系 > 实时观看

### 2.1 核心判断

「让用户实时看到每一代因子的回测结果」对**研究调试**有价值，
对**实盘交易**没有直接价值，且有两个真实害处：

1. **诱导干预**：人盯着中间代结果会忍不住手动挑选"看起来好的"个体，
   这等价于把人脑变成又一层未被校正的多重检验
2. **眼球过拟合**：反复观看同一 OOS 段的中间结果，会让研究者对该段
   数据形成先验，污染后续所有判断

### 2.2 正确形态：三层透明度

| 层 | 内容 | 载体 | 受众与时机 |
|----|------|------|-----------|
| 研究日志 | 每代摘要（fitness/sharpe/dsl）实时流 | SSE → Console | 研究者，调试时看 |
| **审计谱系** | 每个个体的完整评估结果 + 数据哈希 + 种子 + 配置 | 数据库（RunManifest）| 事后回放：因子失效归因、验证门审查、合规 |
| 决策记录 | 状态流转的时间/依据/操作人 | 生命周期状态机日志 | 资本分配复盘 |

**原则：完整细节必须持久化（谱系），但默认不推 UI（观看）。**
实盘中真正被问到的问题是"这个亏钱的因子当初凭什么上线"——
回答它需要的是可回放的谱系，不是当时有没有人盯着屏幕。

## 三、数据引入路线（按泄漏风险与性价比排序）

### 3.1 铁律

任何新数据源接入的前置条件：**带公布时间戳进入 PIT 层**。
财报的生死线是"报告期 ≠ 可见期"——Q3 财报覆盖 9 月 30 日，
但 11 月才公布；用报告期对齐等于内置一个多月的前视泄漏，
产出的"财报因子"回测再好看也是假的。

### 3.2 引入顺序

| 优先级 | 数据类 | 典型因子 | 关键风险 | 引入前提 |
|--------|--------|---------|---------|---------|
| 1 | 财报基本面（盈利/估值/质量）| E/P、盈利修正、应计项 | 公布日对齐；财报重述 | PIT 层就绪（Phase 8）+ DSL 支持稀疏季频字段 |
| 2 | 盈利公告事件 | PEAD（公告后漂移）、SUE | 同上 | 同上 |
| 3 | 资金流 | 北向资金、ETF 申赎、融资融券 | 数据延迟不一致 | 每源独立的可见时间戳 |
| 4 | 期权市场 | IV skew、Put/Call、期限结构 | 流动性差的合约噪声大 | 仅对期权活跃标的 |
| 5 | 另类数据（新闻情感/卫星等）| 事件驱动 | 成本高、历史短、易失效 | 前四类饱和后再考虑 |

### 3.3 DSL 的配套演进

季频基本面字段接入 DSL 需要新的字段语义：**稀疏字段**——
只在公布日后可见，日频面板上从公布日 ffill，公布日前为 NaN。
这是 DSL 从"技术量价语言"升级为"金融结构语言"（原审计点 2）
的实质台阶，比添加更多技术算子更重要。

## 四、第一性原理：Agent 架构的必要性判据

### 4.1 从盈利公式倒推

```
实盘盈利 = (真实 alpha × 可用容量 − 交易成本) × 执行纪律
```

这个公式里**没有**"编排复杂度"这一项。多 agent、planning、
工具链本身不产生 alpha，不扩大容量，不降低成本。历史上量化机构
的失败极少因为"自动化不够复杂"，几乎全部因为：数据有泄漏、
验证不严、成本低估、纪律崩溃。**所以默认答案是：不需要，
除非满足下述判据。**

### 4.1b 概念澄清：Tool-calling ≠ 多 Agent（易混淆，明确区分）

本节反对的是**多 agent 编排**的过早引入，不是反对 tool call。四个层次的立场：

| 层次 | 形态 | 立场 |
|------|------|------|
| 裸 LLM API | 只产文本，不接触工具 | 无用形态——LLM 自身连一次回测都执行不了 |
| **Tool-calling 单 agent** | LLM 做决策编排 + 确定性工具执行 | **现有架构（LangChain agent + 8 个量化工具），正确且必要**；工具库随数据扩展持续增长（如未来 `tool_query_fundamentals`）|
| 多 agent 协作 | 多个独立目标的 agent 相互通信 | 按 §4.2 判据引入，现阶段不需要 |
| LLM 进交易回路 | 运行时决策依赖 LLM | 永久反模式（§4.3-1）|

同时系统必须保持**双通道**：LLM 通道（tool-calling agent，研究体验）与
确定性通道（FallbackOrchestrator / REST API 直调），后者保证零 LLM
可用性——这是 §4.3-3"生产不依赖 AI 在线"的结构保障。

### 4.2 拆分 agent 的唯一正当判据

当一个环节需要与主流程**目标函数对立、信息隔离或时间尺度不同**时，
拆分才有真实回报：

| 角色 | 与主流程的对立性 | 引入时机 | 判据触发条件 |
|------|-----------------|---------|-------------|
| **对抗性审计者（红队）** | 目标相反：生成者想让因子通过，审计者想证伪它（泄漏/容量/拥挤度/经济逻辑漏洞）| **Paper 期（最早有回报的多 agent 角色）** | 候选产出速度 > 人工审查带宽 |
| 数据质量哨兵 | 时间尺度不同：持续监控 vs 批式研究 | 引入第 2 类以上数据源后 | 数据源 ≥ 3 且异常靠人发现 |
| 研究预算分配（planning）| 层级不同：决定"探索哪些假设方向"vs"执行单次探索" | 多市场多假设并行后 | 算力成为瓶颈且方向选择明显影响产出 |
| 归因分析师 | 信息隔离：只看实盘结果不看研究过程，避免确认偏误 | 实盘 6 个月后 | ACTIVE 因子 ≥ 5 个 |

### 4.3 明确的反模式（永久有效）

1. **LLM 进入每日交易决策回路**——每日信号计算、下单、风控必须是
   确定性代码；LLM 只允许出现在研究侧（假设生成、代码编写、报告解读）
2. **用编排复杂度替代验证纪律**——如果单 pipeline 的验证门是松的，
   十个 agent 协作产出的还是垃圾，只是更贵的垃圾
3. **把开发工具当运行时依赖**——Claude Code / skills 是开发加速器
   （写模块、审代码、修 bug），生产调度必须内建（APScheduler），
   系统必须在没有任何 AI 在线的情况下完整运行每日循环
4. **为技术履历引入技术**——任何组件引入前先回答：它改善盈利公式
   的哪一项？改善多少？不能回答就不引入

### 4.4 Tool 与 Skill 的引入路线

#### 4.4.1 Tool 的产生原则

**Tool 是已验证能力的暴露层，不是能力本身。** 一个工具被加入 agent
工具库的前置条件（缺一不可）：

1. 底层确定性能力已实现且有测试覆盖（先能力后包装，禁止把未测试代码包成 tool）
2. LLM 在研究对话中确实需要调用它（有真实使用场景，不为凑数）
3. 权限分级已定：**只读 / 写（需幂等）/ 禁区**

权限铁律：**下单类、资金类、状态机流转类操作永不暴露为 LLM 工具**——
状态流转走人工 PATCH 端点，这是 §4.3-1 反模式在工具层的落实。
（`tool_save_alpha` 写台账可接受：只增不改、可审计、无资金后果。）

#### 4.4.2 Tool 引入时间表（随能力落地同步包装）

| 阶段 | 新工具 | 权限 | 依托的能力 |
|------|--------|------|-----------|
| 现有 | generate_dsl / backtest / gp_optimization / mutate_ast / optuna / interpret_factor / save_alpha 等 8 个 | 读+写台账 | Phase 0-5 全部引擎 |
| Phase 6 | `tool_replay_manifest`（按 manifest 重放并对比）、`tool_query_lineage`（查因子进化谱系）| 只读 | 6.5 RunManifest + gp_lineage |
| Phase 7 | `tool_get_paper_positions` / `tool_get_paper_pnl`（查模拟持仓与净值）、`tool_check_data_health`（数据验收门结果）| **只读** | 7.2 PaperBroker + 7.1 摄取门 |
| Phase 8 | `tool_query_fundamentals`（PIT 财报查询，含公布日语义）、`tool_get_regime`、`tool_red_team_review`（对候选出具反方报告）| 只读 | 8.1 PIT 层 + 4.1 Regime + 8.3 红队 |
| 数据扩展期 | 每接入一类新数据源 → 一个对应查询工具 | 只读 | 该数据源的 PIT 接入 |

规律：**Phase 6 之后新增工具几乎全部只读**——研究 agent 对生产状态
（paper 持仓、调度、谱系）只有观察权，没有操作权。

#### 4.4.3 Skill 的两层含义与各自来源

**产品层 skill = DSL 算子库 + 工具库**（agent 的"技能"）：
- 来源：审计识别的表达力缺口（如 §3.3 稀疏财报字段）+ 新数据类接入
- 加入流程（Task 3.4 已验证的固定管线）：fast_ops 实现 → typed_nodes
  文法注册 → 黄金/冒烟测试 → 前端 Monaco 高亮与补全 → 文档
- 节奏：跟随数据引入路线（§3.2），不单独排期

**开发层 skill = Claude Code 项目技能**（`.claude/skills/`）：
- 来源判据：**同一开发工作流手动执行 ≥3 次即沉淀为 skill**
- 本项目已重复出现、值得沉淀的候选：
  - `/verify-phase`：跑 test_phaseN + 全量回归 + 前端 vitest/tsc，输出验收摘要
  - `/audit-align`：抽查报告声称 vs 代码实态（本项目已做过 3 轮的工作流）
  - `/new-operator`：按 4.4.3 管线脚手架一个新 DSL 算子（实现+注册+测试+前端）
- 引入时机：随时（成本低、无运行时影响）；性质是开发加速器，
  受 §4.3-3 约束——生产每日循环对其零依赖

### 4.5 分阶段结论

```
Phase 6-8（当下）   ：零多 agent。确定性 pipeline + 调度器是正确形态。
Paper 验证期        ：引入红队审计（可以是独立 prompt 的 LLM 审查步骤，
                     不必是常驻 agent）——对每个进入 PAPER 的候选出具
                     "反方报告"存入谱系。
数据扩展期          ：数据质量哨兵。
实盘 6 个月后       ：归因分析师 + 研究预算 planning（若算力成为瓶颈）。
```
